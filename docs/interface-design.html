The two principles of module interface design

Contain
and Declare

Patterns are not rules. They are shapes. This is an attempt to find the geometry underneath — the two principles of module interface design from which most good design follows. What happens inside a module is your business. What crosses its boundary is everyone's.

01 — The patterns

Five patterns that live at
module boundaries

Before asking why patterns work, it helps to see what they actually are. These aren't general software shapes — they're specifically answers to the question of how modules should relate to each other. Each one takes a position on what should cross a boundary and how.

Isolation
Ports & Adapters
Domain knows nothing of infrastructure
The port is the declared interface. The adapter is the contained implementation. Dependency points inward — the domain defines what it needs, infrastructure provides it.
⚠ Interfaces with one implementation — abstraction cost with no modularity payoff
Extensibility
Plugin
Host defines extension points, third parties fill them
Deliberate, sanctioned hole in containment. The host declares what can cross — the extension interface — and third parties implement against it.
⚠ API ossification — once published, the extension interface can never change without breaking everyone
Communication
Observer
Producers don't know consumers
Loose coupling — event sources are isolated from their reactions
⚠ Consumer relationships cross the boundary undeclared. Two streams interacting produce implicit ordering. Rx/observables are the correction — making composition explicit
Composition
Middleware Chain
Wrap before/after without modifying
Bracket arbitrary behavior around a handler with shared scope across the suspension point
⚠ Ordering dependencies are load-bearing and cross the boundary undeclared — invisible constraints that callers must know but can't see
Effects
Algebraic Effects
Capabilities declared, not assumed
Effects accumulate through internal functions and are declared once at the module surface. Callers see the complete dependency set. The compiler enforces it.
⚠ Applied at wrong granularity — every internal function rather than the module surface — becomes Haskell's IO problem

Three of these are directly about declaring what crosses the boundary — observer and middleware by their absence of declaration, algebraic effects by its presence. Ports & adapters is the clearest statement of containment. Plugin is the honest acknowledgment that sometimes containment needs a sanctioned escape hatch.

The question is: what principle explains when each of these holds, and when it breaks?

02 — The failure modes

Patterns fail at the same place

The interesting thing about where these patterns break is that it's not random. Observer breaks when consumer relationships are implicit. Middleware breaks when ordering is implicit. Plugin breaks when the extension interface ossifies. There's a common denominator.

Middleware — where it holds vs where it breaks
Holds: behavior bracketed cleanly
auth
logging
handler
Each middleware wraps the next. Behavior is added without modifying the handler.
Breaks: implicit ordering dependency
compress
auth
handler
↩ auth must run before compression — constraint undeclared

Every pattern has a version of this. Observer's consumer ordering is undeclared. Plugin's extension interface becomes load-bearing the moment it's published. Ports & adapters over-abstracts when there's only ever one adapter.

In each case, the pattern breaks when a dependency that crosses the boundary isn't declared — something callers need to know but can't see from the interface.

Every pattern fails when a dependency crosses the boundary without being declared.
This isn't a coincidence. It's a clue about what the patterns are actually made of.
03 — The boundary

The boundary is specifically a module interface

The principles don't apply to implementation details. What happens inside a module — its internal state, its private functions, its performance choices — is irrelevant. The principles activate only at the surface where modules meet: the interface.

Module boundary — scope of the principles
Inside the module — irrelevant
Mutable local state
Private functions
Internal data structures
Performance choices
Implementation mechanism
Can be as messy as needed
Module interface
At the interface — principles apply
Declared dependencies
Effects / capabilities required
Failure modes
What this module does to the world
Must be honest and complete

This is a precise scope. A function's internal loop variables, a class's private fields, a module's implementation files — none of these are subject to the principles. The principles are about interface design, not code style.

Haskell's ST monad is the canonical example of this done right — internally mutable, externally pure. The module interface is honest about what escapes (nothing). The implementation is free to use mutation for performance. The boundary is the design.

The traditional formulation "separation of concerns" fails because it splits by technical type — controller, service, repository — rather than by module boundary. When a feature changes you touch all three layers. The concern was redistributed, not contained.
At module boundaries: contain what is internal.
Expose what crosses.
This is principle one. Scope is everything — it only applies at interfaces between modules, not inside them. A module can do anything internally as long as its interface honestly declares what escapes.
04 — Effects as module dependencies

Effects accumulate to the boundary,
not every call site

Mutable state, side effects, async, exceptions, resource acquisition — all non-local consequences that cross module boundaries. The question isn't whether to track them, but where. The answer is: at the module interface, and only there.

Effect accumulation — internal vs interface

Database Network Clock
Internal functions carry effects freely. The public interface exposes the union — once, at the boundary.

This is what "declare" means precisely. Not annotating every internal function — that's haskell's mistake. Declaring the complete set of non-local consequences at the module surface, where it's visible to callers and enforceable by the compiler.

But declaring effects at a boundary requires that they be first-class values — something you can name, accumulate, and reason about before they execute. That's reification. And it appears far beyond just type systems.

05 — Reification of intent

Making consequences first-class values

Declaring what crosses a boundary requires a mechanism. You can't declare something you can't name. The mechanism is reification — turning a would-be action into a first-class value that can be inspected, deferred, replayed, or rejected before it touches the world. This pattern appears at every level of the stack.

WAL is the clearest example. Before executing a write that will mutate durable state, serialize the intention into the log first. If the system crashes, the intention survives. Execution becomes idempotent — you can apply the same intention twice without corruption.

WAL as reification under crash conditions
01

Before any mutation, the intent is written durably. This is reification — the action becomes a value.

02

The log entry exists independently of the execution. Crash here — the intent survives. Replay is possible.
03
State is derivative
The log is truth. The state is a projection. Separating what happened from what currently is.

Redux dispatch(action), kafka's append-only log, event sourcing, CRDT operations — all saying the same thing: Record the intent before executing it. Keep the two steps separate.

Declare what crosses the boundary —
Make non-local consequences first-class values.
This is principle two: declare. You can't expose what you can't name. Once it's a value, you can inspect it, defer it, test it, replay it, reject it. The compiler can enforce it. The question that remains is where the boundary actually lives.
06 — The granularity question

Where you draw the boundary is the design

Both principles are scoped to module interfaces. The remaining question is: where exactly is the boundary? And what do you choose to make primitive — intent or state? These are design decisions, and they have dual answers.

WAL and migrations — the reification principle, inverted
WAL / event sourcing
Operation is primitive
Reify intent before execution. State is derived by replaying the log. The log is truth.
Best for: audit, replay, temporal queries
⇄
Duals
Snapshots / migrations
State is primitive
Reify state explicitly. Operations are derived by diffing. The snapshot is truth.
Best for: current state queries, schema evolution
Git does both — commits are full state snapshots, but the DAG gives operations for free by diffing adjacent snapshots. Neither is primitive. This is why git's model is unusually durable.

The duality isn't a contradiction — it's the same principle applied in opposite directions. You choose what to make first-class based on your access pattern. The design decision is about tradeoffs, not correctness.

Granularity — where the principle earns its cost
Too fine — haskell's IO
Applying declaration at every internal function. Every private call site annotated. Implementation details bleed into the interface model. Declaration cost everywhere, modularity payoff nowhere.
Right granularity — module interface
Effects accumulate internally, declared once at the public surface. Opinionated about the 80% case, escape hatches for the rest. The principle applied exactly where it earns its cost.
The principles optimize for modularity — reasoning about a module from its interface without knowing what's behind it. Applying them finer than the module boundary undermines that goal. Callers start needing to understand internal structure.
07 — The two principles

Everything reduces to two things

The five patterns we started with — ports & adapters, plugin, observer, middleware chain, algebraic effects — are all positions on the same two questions: what does the module contain, and what does it declare at its boundary.

Principle one
Principle two
The boundary

expose what crosses
Ports & adapters — contains domain logic, exposes infrastructure dependency

Pure functions — contain all behavior, nothing crosses, nothing declared

The mechanism

WAL — intention is a value before execution
Event sourcing — change is a value, state is derivative

The patterns that falter are the ones where one of these principles is only partially applied at the module interface. Middleware fails because it doesn't declare ordering dependencies. Observer fails because it doesn't declare the consumer relationship. Plugin fails when the declared extension interface can never evolve.

The patterns that hold are the ones where both principles are cleanly applied. Ports & adapters contains implementation, declares the dependency. Algebraic effects declare the complete capability set at the module surface. Pure functions are the trivial case — nothing crosses, nothing needs declaring.

The principles apply at module interfaces — function signatures, public APIs, service contracts. They don't apply inside modules. A system can apply these at function-module-service levels simultaneously because each level has its own interface. The same principles, applied at the right granularity per level.
08 — Vs traditional principles

SOLID and DRY are projections,
not foundations

The traditional software engineering principles — SOLID, DRY, KISS, separation of concerns — aren't wrong. They're low-resolution projections of the two core principles onto specific contexts. They describe the shadow without describing what casts it.

Traditional principle What it observes What it misses The underlying principle
Single Responsibility A class should have one reason to change "reason to change" is undefined — does all the work without precision Things that share a boundary should change together
DRY Don't repeat yourself — deduplicate Optimizes textual deduplication, not conceptual locality. Identical code that changes for different reasons should stay separate Co-locate by change locality, not textual similarity
Separation of Concerns Split things apart Splits by technical type (controller/service/repo), not by locality. The concern was redistributed, not contained Contain by change locality, expose by escape behavior
Dependency Inversion Depend on abstractions, not concretions Closest to the real principle but stops short of the mechanism Expose infrastructure dependency explicitly at the boundary
Open/Closed Open for extension, closed for modification Doesn't explain why or where the extension point should be Contain implementation, reify extension as a declared interface
Law of Demeter Don't reach through objects Rule without explanation — why is reaching through bad? Reaching through crosses a boundary that wasn't declared

The traditional principles are useful as communication tools. "This violates SRP" is a useful shorthand in a code review even when the underlying reasoning is imprecise. But they fail as design tools because they're observational — here's what good code looks like — rather than generative — here's how to reason toward it.

You can follow all of SOLID and still produce bad architecture. Because SOLID doesn't tell you where the real boundaries are — only how to respect them once you've found them.

The traditional principles describe symptoms.
These two describe the cause.
Once you have the cause, you can reason toward good design from first principles — not just recognize it after the fact. And you can diagnose precisely where a pattern breaks down, rather than having a vague sense that something feels wrong.
In summary
I
Contain — hide what is internal to the module.
II
Declare — make non-local consequences first-class at the interface.
These principles are specifically about module interface design — not internal implementation. A module can do anything inside. What it declares at its boundary must be honest and complete. The adversarial conditions change across levels and domains. The principles don't.
