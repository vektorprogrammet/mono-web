Domain-First SDK Design

A typed SDK layer over the Symfony/API Platform backend — decoupled from the OpenAPI spec, scoped to Vektorprogrammet's domain, and ready for the backend migration to TypeScript.

This design didn't emerge from a checklist. It's the result of a few ideas from theory that happen to reinforce each other. If any of these sound interesting, go learn them — it's worth it.

  • DDDDomain-driven design — shape the interface around the business domain, not the database or API. the ubiquitous language of the organisation becomes the language of the code.
  • P&APorts and adapters (hexagonal architecture) — the domain defines the port, infrastructure conforms to it. dependency inversion is the mechanism: the SDK owns the contract, the adapter implements it, never the other way around.
  • TTType theory — invalid states become verifiably illegal. a Bergen client cannot call Trondheim operations not because of a runtime guard, but because the method doesn't exist on the type.
  • FCISFunctional core, imperative shell — pure domain logic at the centre, effects and mutation pushed to the edges. the adapter is where IO lives; the domain never is.

Together they make the semantics explicit and the mechanics implicit — dependencies, effects, errors, and state transitions are all visible in the types, while the how of talking to a server is hidden behind the abstraction where it belongs.

SDK Public Interface
plain promises · typed errors · scoped client
▼

The contract the apps depend on. Exposes domain operations shaped around Vektorprogrammet's workflows, not HTTP endpoints. Computed fields like Application status are resolved here — consumers never see raw PHP integer constants. The client type encodes role, department, and team scope.

// domain-shaped operations
sdk.interviews.schedule(id, { datetime, room, interviewer })   // ✓ domain operation
sdk.interviews.put(id, { interviewStatus: 0 })                 // ✗ raw PHP integers leak

// application status is a string enum, not a PHP int constant
application.status  // "received" | "invited" | "accepted" | "completed" | "assigned" | "cancelled"
application.status  // APPLICATION_RECEIVED(1), INVITED_TO_INTERVIEW(2)...

// typed result — explicit errors
const result = await sdk.receipts.approve(id)
if (!result.ok) {
  result.error // { type: "not_pending" | "unauthorized" | "not_found" }
}

Effect-TS Internals
typed errors · context · deferred execution
▼

Hidden from consumers entirely. Handles auth state, request batching, structured concurrency, and typed error propagation. The public interface wraps Effects in plain promises — apps never see Effect types. If a future maintainer wants to replace Effect-TS, they reimplement the public interface; no app code changes.

// internal: Effect pipeline — never exposed to consumers
const listApplications = (params: ListParams) =>
  Effect.gen(function*() {
    const auth = yield* AuthContext
    const raw  = yield* fetchApplications(auth.token, params)
    return raw.map(deriveStatus)  // integer → string enum
  })

// public surface: plain promise
applications.list = (params) => Effect.runPromise(listApplications(params))

Adapter
symfony now · @monoweb/api later
▼

The only layer that changes during migration. Today it maps domain calls to Symfony's /api/admin/* endpoints, handles Hydra collections, strips IRI references, and normalises integer status constants. When @monoweb/api is ready, a new adapter replaces this one — everything above is untouched.

// Symfony adapter: strip Hydra, resolve IRIs, normalise status
const fetchApplications = (token: string, params: ListParams) =>
  Effect.tryPromise(() =>
    fetch(`/api/admin/admission_periods/${params.admissionPeriodId}/applications`, {
      headers: { Authorization: `Bearer ${token}` }
    })
    .then(r => r.json())
    .then(json => json["hydra:member"].map(normalise))
  )

// @monoweb/api adapter: clean JSON, no translation needed
const fetchApplications = (token: string, params: ListParams) =>
  Effect.tryPromise(() =>
    apiClient.applications.list(params)  // returns Application[] directly
  )

// client type encodes all three dimensions at login
type Client<
  R extends Role,
  D extends Department,
  T extends Team
> = ScopedClient<R, D, T>

// Bergen · Member · Recruitment — cannot touch Trondheim or Accounting
const sdk: Client<Role.Member, Dept.Bergen, Team.Recruitment> = createClient(session)

sdk.applications.list()         // ✓ scoped to Bergen automatically
sdk.receipts.approve()          // ✗ compile error — Accounting team only
sdk.interviews.schedule()       // ✓ available to Recruitment

why this matters

The current PHP server has 12+ guard downgrades where operations that should require ROLE_TEAM_LEADER only check ROLE_TEAM_MEMBER, and 5+ domains with no department scoping at all. The SDK's structural approach means these classes of bugs are caught at compile time rather than discovered in production — a Trondheim client literally cannot call Bergen operations because the method doesn't exist on the type.

sdk responsibility

The PHP backend stores APPLICATION_RECEIVED(1), INVITED_TO_INTERVIEW(2), etc. as integer constants. The SDK adapter runs deriveStatus() — the same top-to-bottom logic from ApplicationManager::getApplicationStatus() — and surfaces "received" | "invited" | "accepted" | "completed" | "assigned" | "cancelled" to the apps. When the backend migrates to TypeScript and starts returning computed status directly, only the adapter changes.

Core Principles

The six decisions that make this architecture work.

01
Domain first, spec second
The SDK interface is shaped around Vektorprogrammet's workflows — admission, recruitment, team management, accounting. The OpenAPI spec is a reference for writing the adapter, not the source of truth for the interface.
02
Illegal states unrepresentable
A Bergen client can't call Trondheim operations. A Member client doesn't have schedule(). An Accounting client has no interviews namespace. Structural absence is stronger than runtime guards — the compiler catches it.
03
Hide implementation artifacts
PHP integer constants, Hydra collections, IRI references, JSON-LD context, hyphenated URL conventions — all stripped at the adapter boundary. Apps work with clean domain types and string enums.
04
One seam, one swap
The adapter is the only thing that changes when @monoweb/api replaces Symfony. All app code, the SDK interface, and the Effect internals are untouched. No big bang rewrite risk.
05
Effect-TS is an implementation detail
Apps see plain promises. If a future maintainer wants to replace Effect-TS, they reimplement the public interface. No lock-in, and the team doesn't need to understand Effect to use the SDK.
06
Explicit errors
Every mutation returns a discriminated union. receipts.approve() can fail with "not_pending", "unauthorized", or "not_found" — each typed. Silent failures are not possible.

Migration Plan

How the Symfony → TypeScript backend switch happens with zero app-side rewrites.

now

Establish the domain interface

Design the SDK interface from Vektorprogrammet's domain, not from the OpenAPI spec. Insert the boundary before apps depend on generated shapes. Even if operations are 1:1 passthroughs internally, the seam exists.

v1

Symfony adapter — full feature parity

Adapter maps domain calls to /api/admin/* endpoints. Handles Hydra, IRIs, integer status constants. Homepage and dashboard build against the domain interface only. Computed status derivation lives here.

→

TypeScript server adapter in parallel

Write a second adapter targeting @monoweb/api. Feature-flag or deploy both simultaneously. No breaking changes to app code — they still call sdk.applications.list().

v2

Cutover + major version bump

Symfony backend deprecated. New features added, removed features dropped. Apps bump the SDK version — call sites are identical. The interface was the contract all along.
