SDK Public Interface
plain promises · typed errors · scoped client
▼

The contract the apps depend on. Exposes domain operations shaped around Vektorprogrammet's workflows, not HTTP endpoints. Computed fields like Application status are resolved here — consumers never see raw PHP integer constants. The client type encodes role, department, and team scope.

// domain-shaped operations
sdk.interviews.schedule(id, { datetime, room, interviewer })   // ✓ domain operation
sdk.interviews.put(id, { interviewStatus: 0 })                 // ✗ raw PHP integers leak

// application status is a string enum, not a PHP int constant
application.status  // "received" | "invited" | "accepted" | "completed" | "assigned" | "cancelled"
application.status  // APPLICATION_RECEIVED(1), INVITED_TO_INTERVIEW(2)...

// typed result — explicit errors
const result = await sdk.receipts.approve(id)
if (!result.ok) {
  result.error // { type: "not_pending" | "unauthorized" | "not_found" }
}

Effect-TS Internals
composition · typed errors · concurrency
▼

Fully internal. Handles request composition, parallel fetching, rich typed errors, and deferred execution. Never exposed to app consumers. Auth state, department scope, and JWT management all live here as Effect services.

// application status derivation as a pure effect
const deriveStatus = (app: RawApplication): ApplicationStatus => {
  if (app.user.isActiveAssistant) return "assigned"
  if (app.user.hasBeenAssistant || app.interview?.interviewed) return "completed"
  if (!app.interview) return app.admissionPeriod ? "received" : "not_received"
  return mapInterviewStatus(app.interview.interviewStatus)
}

// JWT refresh as an effect — transparent to callers
const authedRequest = Effect.gen(function* () {
  const auth = yield* AuthService
  const token = yield* auth.getValidToken()  // refreshes if expired
  return yield* HttpService.withToken(token)
})

Adapter
swappable · maps domain → http
▼

Translates domain calls to concrete HTTP requests. This is the only layer that changes during the backend migration. It knows about API Platform's naming conventions, Hydra pagination, IRI references, and JSON-LD — none of that leaks up.

// today — API Platform adapter
// handles: Hydra collections, IRI refs like "/api/users/42", hyphen URLs
const symfonyAdapter = {
  listApplications: ({ admissionPeriodId }) =>
    get(`/api/admin/applications?admissionPeriod=${admissionPeriodId}`)
      .then(fromHydraCollection)   // strips Hydra wrapper
      .then(apps => apps.map(normalizeIris))   // "/api/users/42" → 42
}

// tomorrow — TypeScript server adapter, zero changes above
const tsAdapter = {
  listApplications: ({ admissionPeriodId }) =>
    get(`/api/applications?period=${admissionPeriodId}`)
      .then(r => r.data)   // plain JSON, no Hydra
}
↓

Permission Model

Three orthogonal dimensions. A user's effective access is their intersection: role hierarchy × department × team. Each dimension is independent.

Hierarchical — each role inherits everything below it. Assistant < Member < Leader < Admin. Some operations require runtime role checks since roles come from the server JWT.

sdk implication

Department scope is baked into the authenticated client. All queries are automatically filtered to the user's department. A Bergen user calling sdk.applications.list() only ever sees Bergen applications — there's no department parameter to accidentally forget. Admins get an optional override.

This also fixes a known issue in the current PHP server: AdminApplicationListProvider, AdminUserListProvider, and several others have no department scoping at all — any team member can query any department's data.

Teams are functional groups within a department. Each team has its own membership, responsibilities, and operations. Team membership is tracked across semesters with suspension and leader flags.

membership state

Membership has two independent dimensions: isActive() (temporal — within start/end semester) and isSuspended. These are separate checks — isActive() does NOT account for suspension. The SDK surfaces both as distinct fields so the UI can show "paused" vs "former member" correctly, without the caller having to understand the underlying implementation.

Effective access is the intersection of role × department × team. The client type is fully scoped at login — invalid operations are structurally absent, not just runtime-forbidden.

// login returns a scoped client
const result = await sdk.auth.login({ email, password })
// Client<Role.Member, Dept.Bergen, Team.Recruitment>

// intersection: Member + Bergen + Recruitment
client.applications.list()            // ✓ Bergen applications only, auto-scoped
client.interviews.schedule(id, data)  // ✗ not on Member type (Leader+ only)
client.receipts.approve(id)           // ✗ not on Recruitment team type
client.applications.list({ dept: "Trondheim" })
                                       // ✗ Bergen client can't cross-scope

// Leader in Accounting at Trondheim — completely different type
// Client<Role.Leader, Dept.Trondheim, Team.Accounting>
leaderClient.receipts.approve(id)     // ✓
leaderClient.interviews.schedule()   // ✗ not on Accounting team type
leaderClient.applications.list()     // ✗ different dept

why this matters

The current PHP server has 12+ guard downgrades where operations that should require ROLE_TEAM_LEADER only check ROLE_TEAM_MEMBER, and 5+ domains with no department scoping at all. The SDK's structural approach means these classes of bugs are caught at compile time rather than discovered in production — a Trondheim client literally cannot call Bergen operations because the method doesn't exist on the type.

sdk responsibility

The PHP backend stores APPLICATION_RECEIVED(1), INVITED_TO_INTERVIEW(2), etc. as integer constants. The SDK adapter runs deriveStatus() — the same top-to-bottom logic from ApplicationManager::getApplicationStatus() — and surfaces "received" | "invited" | "accepted" | "completed" | "assigned" | "cancelled" to the apps. When the backend migrates to TypeScript and starts returning computed status directly, only the adapter changes.

Core Principles

The six decisions that make this architecture work.

01
Domain first, spec second
The SDK interface is shaped around Vektorprogrammet's workflows — admission, recruitment, team management, accounting. The OpenAPI spec is a reference for writing the adapter, not the source of truth for the interface.
02
Illegal states unrepresentable
A Bergen client can't call Trondheim operations. A Member client doesn't have schedule(). An Accounting client has no interviews namespace. Structural absence is stronger than runtime guards — the compiler catches it.
03
Hide implementation artifacts
PHP integer constants, Hydra collections, IRI references, JSON-LD context, hyphenated URL conventions — all stripped at the adapter boundary. Apps work with clean domain types and string enums.
04
One seam, one swap
The adapter is the only thing that changes when @monoweb/api replaces Symfony. All app code, the SDK interface, and the Effect internals are untouched. No big bang rewrite risk.
05
Effect-TS is an implementation detail
Apps see plain promises. If a future maintainer wants to replace Effect-TS, they reimplement the public interface. No lock-in, and the team doesn't need to understand Effect to use the SDK.
06
Explicit errors
Every mutation returns a discriminated union. receipts.approve() can fail with "not_pending", "unauthorized", or "not_found" — each typed. Silent failures are not possible.

Migration Plan

How the Symfony → TypeScript backend switch happens with zero app-side rewrites.

now

Establish the domain interface

Design the SDK interface from Vektorprogrammet's domain, not from the OpenAPI spec. Insert the boundary before apps depend on generated shapes. Even if operations are 1:1 passthroughs internally, the seam exists.

v1

Symfony adapter — full feature parity

Adapter maps domain calls to /api/admin/* endpoints. Handles Hydra, IRIs, integer status constants. Homepage and dashboard build against the domain interface only. Computed status derivation lives here.

→

TypeScript server adapter in parallel

Write a second adapter targeting @monoweb/api. Feature-flag or deploy both simultaneously. No breaking changes to app code — they still call sdk.applications.list().

v2

Cutover + major version bump

Symfony backend deprecated. New features added, removed features dropped. Apps bump the SDK version — call sites are identical. The interface was the contract all along.
